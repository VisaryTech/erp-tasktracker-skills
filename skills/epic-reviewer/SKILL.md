# Проверка Эпика

## Входные данные
- Текст эпика.

## Принципы Проверки

- Не додумывать требования за бизнес.
- Любой недостающий или расплывчатый пункт считать пробелом.
- Фиксировать противоречия между разделами.
- Быть критичным к измеримости, проверяемости и полноте формулировок.
- Не переходить к проектированию решения, архитектуры или реализации, пока эпик не готов.

## Процедура

1. Считать и кратко нормализовать содержание эпика (1-3 предложения, без интерпретаций).
2. Пройти чеклист по всем разделам ниже.
3. Для каждого пункта определить статус:
- `OK`: информация есть и формулировка проверяема.
- `GAP`: информации нет или недостаточно.
- `RISK`: есть риск, который может сорвать оценку или реализацию.
- `CONFLICT`: есть противоречие с другими пунктами эпика.
4. Сформировать итоговую готовность:
- `READY`, если нет `GAP` и `CONFLICT` по критическим пунктам.
- `NOT_READY`, если есть хотя бы один критический пробел или конфликт.
5. Вернуть результат строго в формате из раздела "Формат Результата".

## Матрица критичности

По умолчанию считать критическими пункты, без которых команда не может безопасно оценить объем и принять работу в разработку:

| Раздел | Критичные пункты |
| --- | --- |
| Цели и контекст | бизнес-проблема, KPI, границы scope, зависимости |
| Scope и функциональные требования | ограничения релиза, MVP/минимальный запуск |
| Процессы и бизнес-логика | затрагиваемые роли и системы (если влияет на процесс) |
| Данные | входные/выходные данные, источники, требования по ПДн/регуляторике |
| Интеграции | внешние/внутренние системы, направления обмена, API-ограничения |
| Нефункциональные требования | безопасность, надежность/SLA (если релиз production) |

Если пункт не влияет на возможность безопасной оценки и старта разработки, помечать как `NON_CRITICAL`.

## Чеклист Проверки

### 1. Цели и контекст
- Понятна бизнес-проблема, которую решает эпик.
- Сформулирована измеримая цель или KPI.
- Эпик решает одну бизнес-проблему.

### 2. Scope и функциональные требования
- Описан общий сценарий использования.
- Зафиксированы ключевые пользовательские кейсы.
- Есть список high-level требований.
- Нет противоречий между требованиями.

### 3. Процессы и бизнес-логика
- Описан текущий процесс as-is при необходимости.
- Описан целевой процесс to-be.
- Понятны изменения в операционных процессах.
- Выявлены затрагиваемые роли и системы.

### 4. Данные (если есть)
- Определены входные данные.
- Определены выходные данные.
- Понятны источники данных.

### 5. Интеграции (если есть)
- Определены внешние и внутренние системы.
- Понятны направления обмена данными.
- Зафиксированы ограничения по API.
- Есть понимание контрактов и форматов.

### 6. Нефункциональные требования
- Производительность.
- Надежность и отказоустойчивость.
- Безопасность.
- Логирование и аудит.
- Масштабируемость.
- Ограничения по SLA.

### 7. UX и фронтовая часть
- Есть описание пользовательских сценариев.
- Понятны роли и права доступа.

## Формат Результата

Отвечать в `Markdown`, строго в следующем виде:

- `Готовность: READY | NOT_READY`
- `Критические блокеры: <число>`
- `Разделов с замечаниями: <число>`

- `Замечания:`
  - `<Раздел 1>`
    - `Тип: GAP | RISK | CONFLICT`
    - `Критичность: CRITICAL | NON_CRITICAL`
    - `Проблема: <что не так>`
    - `Нужно уточнить: <что именно требуется>`
    - `Подтверждение: <цитата/фрагмент>`
  - `<Раздел 2>`
    - `...`

- `Что нужно закрыть для READY:`
  - `<условие 1>`
  - `<условие 2>`
  - `<условие 3>`

Если замечаний нет, явно написать: `Пробелы, противоречия и критические риски не выявлены`.

## Запреты

- Не предлагать реализацию, архитектуру, декомпозицию задач, оценки или roadmap до устранения критических пробелов.
- Не заменять формальные замечания общими советами.
- Не смягчать формулировки при явной неполноте или невалидности входных требований.
